# `anonId` Has No Format Validation — Enables Cross-User Subscription Theft via localStorage Injection

## Problem Statement

`claimSubscription` accepts `anonId` as a raw `v.string()` with no format validation. An attacker who can write to `localStorage` (via XSS, a compromised browser extension, or same-origin script) can inject any string — including another user's Clerk ID — as `wm-anon-id`. When the victim signs in, `claimSubscription` will run a database query for that injected value, and if it matches records owned by the victim's target, those records are migrated.

## Findings

**Attack scenario:** `localStorage.setItem('wm-anon-id', 'user_target_clerk_id')`. When the victim signs in, `src/App.ts:803` calls `client.mutation(api.payments.billing.claimSubscription, { anonId: 'user_target_clerk_id' })`. The mutation queries subscriptions/entitlements for `userId === anonId`, and if such records exist, migrates them to the now-authenticated victim, stripping the target user's entitlements.

The `wm-anon-id` is generated by `crypto.randomUUID()` and has a well-defined UUID v4 format: `/^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i`. Clerk user IDs have the format `user_[alphanum]` — structurally distinct from UUID v4. A format guard in the Convex mutation would reliably reject injected Clerk IDs.

Additionally, `args.anonId === realUserId` is not checked, meaning a user who passes their own Clerk ID as `anonId` could trigger the merge logic on their own account, leading to potential DB transaction errors if two rows with the same user ID are found.

Agent that identified this: security-sentinel (BLOCKING).

## Proposed Solutions

### Option 1: UUID v4 format validation in the Convex mutation (recommended)

Add a format guard at the top of `claimSubscription`, before the rate-limit check:

```ts
const UUID_V4_REGEX = /^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i;
if (!UUID_V4_REGEX.test(args.anonId)) {
  // Not a valid anon ID — could be an injected Clerk ID or garbage
  return { claimed: { subscriptions: 0, entitlements: 0, customers: 0, payments: 0 } };
}
// Guard against self-claim
if (args.anonId === realUserId) {
  return { claimed: { subscriptions: 0, entitlements: 0, customers: 0, payments: 0 } };
}
```

**Pros:** Server-side, cannot be bypassed from client; silent no-op for invalid IDs (no error revealed to attacker)
**Cons:** If the anon ID format changes in future, the regex needs updating

**Effort:** Small
**Risk:** Low

### Option 2: Convex v.string() with custom validator
Use a Convex `v.string()` with a refinement. Convex validators don't support regex refinements natively, so this would require the same manual check inside the handler — equivalent to Option 1.

**Effort:** Small
**Risk:** Low

## Recommended Action

Option 1. Add both the UUID format guard and the `anonId === realUserId` guard at the top of the handler.
